# S40.1 Design: Public Repo Safety Gate

## Complexity

Moderate.

This story touches repository policy, public documentation, local runtime conventions, and manual verification. It does not introduce executable product code or external integrations.

## Problem

Ahuehuete Kokoro is intended for public sharing, but the current repo boundary does not fully protect local runtime state, guest exports, generated reports, platform account mappings, or future RaiSE session artifacts.

## Value

After this story, Eduardo can share the repo with much less risk of leaking private guest data or credentials. Future orchestrators can reference a clear privacy contract instead of inventing safety rules per skill.

## Approach

Create a baseline public-safety layer:

- Expand `.gitignore` with private workspace, exports, generated reports, media, and local RaiSE runtime paths.
- Strengthen `kokoro-privacy-protocol.md` with public/private boundaries, platform ID rules, and MCP credential handling.
- Add `kokoro-share-readiness.md` as the manual checklist before sharing or publishing.
- Verify with `git check-ignore`, `git status`, and a regex-based secret scan.

## Components Affected

| Component | Change Type | Purpose |
|---|---|---|

| `.claude/knowledge/kokoro-privacy-protocol.md` | modify | Clarify durable privacy policy |

### Secret Scan Command

```bash
rg -n --hidden --glob '!/.git/**' --glob '!/.agents/**' --glob '!/.claude/skills/**' \
  '(sk-[A-Za-z0-9_-]{20,}|AIza[0-9A-Za-z_-]{20,}|EAA[A-Za-z0-9]{20,}|access[_-]?token|refresh[_-]?token|client[_-]?secret)' .
```

Expected result: only placeholders and documentation examples, no real credential values.

## Acceptance Criteria

### MUST

- `.gitignore` covers private Kokoro and RaiSE runtime artifacts.
- Privacy protocol describes public repo versus private workspace clearly.
- Share-readiness checklist exists and is usable before inviting collaborators.
- Secret scan finds no real credentials introduced by this story.
- Story progress can be tracked in E40 scope.

### SHOULD

- The checklist should be understandable by a non-technical business operator.
- Examples should use `cliente_NN` and placeholder IDs only.

### MUST NOT

- Do not add real guest names, real ad account IDs, tokens, exports, reports, or screenshots.
- Do not make MCP connectivity a requirement for installing the repo.
- Do not ignore public story/design artifacts needed by RaiSE.

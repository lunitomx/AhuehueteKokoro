# S40.1 Story: Public Repo Safety Gate

## User Story

As a founder or marketing leader sharing Ahuehuete Kokoro publicly,
I want the repository to prevent sensitive guest data from being committed,
so that Kokoro can be distributed with confidence before any platform or guest workflow is used.

## Acceptance Criteria

### AC1: Runtime State Is Ignored

Given a local Kokoro workspace with `.kokoro/`, `.raise/`, `.rai/`, `.env`, exports, reports, and generated media,
when a contributor checks `git status`,
then those private/runtime artifacts are ignored unless intentionally whitelisted.

### AC2: Public Privacy Protocol Exists

Given a new contributor reads the repo,
when they need to understand where guest data belongs,
then a public privacy protocol explains what may live in the repo and what must stay in a private workspace.

### AC3: Share Readiness Checklist Exists

Given Eduardo wants to share Kokoro with a guest or collaborator,
when he runs the checklist manually,
then it covers secrets, guest names, platform IDs, runtime state, generated reports, and MCP credential boundaries.

### AC4: No Real Secret Is Introduced

Given this story changes public files,
when the repository is scanned for obvious token patterns,
then no real API key, access token, refresh token, or guest account credential is present.

## Examples

### Ignored Runtime Files

```text
.env
.kokoro/clients.json
.raise/rai/personal/real-clients.txt
exports/cliente_01/google-ads/search-terms.csv
reports/cliente_01/weekly-pulse.md
generated/cliente_01/creative.png
```

These files should be local-only.

### Allowed Public Files

```text
.claude/commands/kokoro-google-ads-run.md
.claude/knowledge/kokoro-privacy-protocol.md
work/epics/e40-kokoro-operability-orchestrators/scope.md
```

These files can describe methods, placeholders, and fake examples.

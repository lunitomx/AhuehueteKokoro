# S40.3 Design: Orchestrator Contract

## Complexity

Moderate.

The story defines a reusable operating contract that several later stories will depend on. It does not add runtime code or connect external services.

## Problem

Kokoro now has an executive router, but the planned operating runs still need a shared contract. Without one, each orchestrator could invent its own gates, skip MCP health, expose private data, or blur recommendations with account mutations.

## Value

Future Kokoro runs can be built faster and more consistently. A business user sees the same shape every time: objective, gates, executive read, recommendations, and explicit permission before action.

## Approach

- Create `.claude/knowledge/kokoro-orchestrator-contract.md`.
- Define phases: invitation, context, runtime/MCP, data plan, knowledge load, analysis, recommendation, permission, close.
- Define gates and allowed statuses.
- Define output template.
- Define fallback behavior when data/MCP is unavailable.
- Define privacy and mutation boundaries.
- Reference the contract from `kokoro-executive-router.md`.

## Examples

### Standard Gate Statuses

```text
Pass     - completed with enough evidence
Partial  - some evidence exists, but not enough for full confidence
Blocked  - required condition missing; do not proceed with data-backed claim
Skipped  - not needed for this route
```

### Standard Output

```markdown
## {Run Name} — {Guest or Context}

| Gate | Status | Evidence |
|---|---|---|
| Context resolved | Pass | Invitado: cliente_01 |
| MCP health | Blocked | google-ads not connected |
| Data completeness | Skipped | no platform data queried |
| Knowledge loaded | Pass | kokoro-mcp-google-ads.md |
| Privacy check | Pass | no private data persisted |

### Executive Read
{interpretation}

### What I Would Not Touch Yet
{patience/no-action guidance}

### Recommended Next Actions
{ranked actions}

### Permission Gate
{explicit invitation before mutation}
```

## Acceptance Criteria

### MUST

- Contract defines all core phases.
- Contract defines reusable gates and statuses.
- Contract defines fallback when MCP/data is unavailable.
- Contract defines privacy and mutation boundaries.
- Executive router references the contract.

### SHOULD

- The contract should be readable by a business operator.
- Later orchestrators should be able to copy its output template directly.

### MUST NOT

- Do not create empty command files for future orchestrators.
- Do not imply MCPs are connected by default.
- Do not include real guest or platform data.

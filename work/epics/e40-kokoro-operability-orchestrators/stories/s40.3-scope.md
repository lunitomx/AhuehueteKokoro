# S40.3 Scope: Orchestrator Contract

## Objective

Create the shared contract that future Kokoro operating runs must follow: phases, gates, status language, output template, fallback behavior, privacy handling, and explicit permission before action.

## In Scope

- Add a knowledge file defining the Kokoro orchestrator contract.
- Define reusable gates and allowed statuses.
- Define a standard output shape for executive runs.
- Define fallback behavior when MCP or data is missing.
- Define permission boundaries for mutating actions.
- Reference the contract from the executive router.

## Out of Scope

- Implementing any specific orchestrator command.
- Connecting MCP servers.
- Creating executable validation tooling.
- Changing every existing skill.
- Creating hooks or CI.

## Done Criteria

- `.claude/knowledge/kokoro-orchestrator-contract.md` exists.
- Contract includes phases, gates, statuses, output template, fallback behavior, privacy, and permission rules.
- `kokoro-executive-router.md` references the contract for operating routes.
- Contract uses business-readable language.
- Verification confirms no credentials or private guest data were introduced.

## Risk Notes

- A contract that is too long will be ignored; keep the top-level structure easy to scan.
- A contract that is too vague will not prevent skipped MCP/privacy gates.
- This story should not prematurely implement the specific Google Ads run.

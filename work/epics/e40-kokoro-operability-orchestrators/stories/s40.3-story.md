# S40.3 Story: Orchestrator Contract

## User Story

As Eduardo building Kokoro's operating layer,
I want every future orchestrator to follow one shared contract,
so that Google Ads, weekly marketing, launch, acquisition, and share-readiness runs behave consistently and do not skip gates.

## Acceptance Criteria

### AC1: Shared Contract Exists

Given a future E40 orchestrator is created,
when the author needs its structure,
then a shared contract defines phases, gates, output format, and permission boundaries.

### AC2: Gates Are Reusable And Business-Readable

Given a business user sees an orchestrator response,
when Kokoro reports progress,
then gate names and statuses are understandable without reading implementation details.

### AC3: Data-Backed Runs Cannot Skip MCP And Privacy Gates

Given an orchestrator depends on platform data,
when MCP is not registered or unhealthy,
then the contract requires a graceful fallback instead of invented data.

### AC4: Action Requires Explicit Invitation

Given an orchestrator can mutate an account or artifact,
when it reaches the recommendation stage,
then the contract requires a permission gate before any action.

## Examples

### Gate Table

```markdown
| Gate | Status | Evidence |
|---|---|---|
| Context resolved | Pass | Invitado: cliente_01 |
| MCP health | Blocked | google-ads not connected |
| Data completeness | Skipped | Data-backed analysis not available |
| Knowledge loaded | Pass | kokoro-mcp-google-ads.md |
| Privacy check | Pass | no private data persisted |
```

### Permission Gate

```markdown
Recomendacion: agregar negativos para terminos irrelevantes.

Permiso requerido:
¿Quieres que prepare la lista para revision o que la subamos a Google Ads?
No ejecuto cambios en la cuenta sin tu invitacion explicita.
```

# S40.7 Design: Launch Run

## Complexity

Complex.

The implementation is a command contract, but launch decisions are high-risk. A bad run could push users into copy, ads, or landing work before the creation has a clear audience, validated promise, primary metric, or tracking.

## Problem

Users say "quiero lanzar" but may be missing upstream clarity. Existing skills cover pieces of the launch lifecycle, yet the user has to know whether to start with canvas, forces, PESCAR, experiment, launch copy, landing, tracking, or scorecard.

## Value

Kokoro becomes a launch operator: it identifies readiness, chooses the right next method, and prevents premature publication.

## Approach

Create `/kokoro-launch-run` as the business-facing run. It orchestrates tactical skills in a readiness-based sequence:

1. `/kokoro-canvas`
2. `/kokoro-forces`
3. `/kokoro-pescar`
4. `/kokoro-experiment`
5. `/kokoro-launch`
6. `/kokoro-landing`
7. `/kokoro-tracking-check`
8. `/kokoro-weekly-marketing-run`

## Launch Readiness Model

| Gate | Meaning |
|---|---|
| Creation defined | What is being shared is clear |
| Audience clear | The invited person is specific |
| Promise true | The offer/event reality matches the claim |
| Forces known | Trigger, push, inertia, and friction are understood |
| Strategy selected | PESCAR channel/content/CTA direction exists |
| Validation evidence | Experiment or equivalent proof exists |
| Metric defined | One primary metric and thresholds are set |
| Landing path | Landing or conversion path exists or is planned |
| Tracking ready | Measurement can be trusted or blocker is known |
| Readback rhythm | Weekly readback path exists |

## Acceptance Criteria

### MUST

- Create `/kokoro-launch-run`.
- Include a launch readiness table.
- Include a method chain and fallback for each blocked gate.
- Include primary metric, success threshold, invalidation threshold, and readback cadence.
- Include asset plan, landing plan, and tracking plan.
- Include permission boundaries for publishing, saving, campaign creation, landing edits, and tracking changes.
- Use public-safe placeholders only.

### SHOULD

- Keep tactical details delegated to existing commands.
- Clearly distinguish "ready to launch" from "ready to prepare".
- Include a demo path without private platform data.

### MUST NOT

- Generate launch assets when validation/tracking gates are blocked without naming the risk.
- Save private launch plans in public repo paths.
- Ask for credentials in chat.
- Publish or mutate anything without explicit invitation.

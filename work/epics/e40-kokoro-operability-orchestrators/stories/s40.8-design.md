# S40.8 Design: Acquisition Run

## Design Goal

Create `/kokoro-acquisition-run` as the executive workflow for improving acquisition without assuming the answer is "more ads".

The run should help a founder, marketing director, or sales leader answer:

- Is the constraint traffic, activation, offer, landing, tracking, CRM/follow-up, or retention?
- Which part of the system deserves the next investment?
- What should not be touched until evidence is clearer?

## Proposed Shape

Create `/kokoro-acquisition-run` as a business-facing operating run. It sequences:

1. Orchestrator contract and context resolution.
2. Acquisition readiness.
3. Data/MCP and tracking gates.
4. Customer Factory map.
5. Funnel Consciente map.
6. Oferta Mafia and landing review.
7. Bottleneck diagnosis.
8. Action plan with permission boundaries.
9. Readback path through weekly marketing rhythm.

## Decisions

### D1: Make Acquisition A System Run

Acquisition is not only traffic. It includes the moment a person discovers the creation, understands the offer, lands on a page, leaves a signal, enters follow-up, and eventually creates revenue.

### D2: Bottleneck Before Channel

The run should not recommend new ads, budget changes, or channel expansion until it knows whether the constraint is upstream or downstream.

### D3: CRM and Follow-Up Are Gates

For a sales/marketing leader, acquisition is incomplete if leads arrive but nobody sees, qualifies, or follows up on them.

## Risks

| Risk | Mitigation |
|---|---|
| Run becomes too broad | Keep output centered on one active bottleneck and next 3 actions. |
| Missing data leads to speculation | Mark data gates as Pass, Partial, Blocked, or Skipped. |
| Private CRM/report data enters repo | Explicit privacy boundary and no tracked private outputs. |
| The run duplicates weekly pulse | Weekly pulse reports what changed; acquisition run diagnoses what to fix. |

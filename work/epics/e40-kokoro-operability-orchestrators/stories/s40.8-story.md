# S40.8 Story: Acquisition Run

## User Story

As a founder, marketing director, or sales leader,
I want Kokoro to diagnose acquisition as a full system,
so that I do not ask for more traffic when the real constraint is offer clarity, activation, landing conversion, tracking, or follow-up.

## Context

Kokoro already has tactical skills for Customer Factory, Funnel Consciente, Oferta Mafia, landing evaluation, analytics, tracking, scorecards, and platform-specific ads. S40.8 creates the operating run that sequences those pieces into one executive acquisition workflow.

This run is for the leader asking:

- "Como consigo mas invitados?"
- "Por que llegan leads pero no cierran?"
- "Que parte del embudo esta rota?"
- "Donde debo invertir la siguiente semana?"
- "Necesito mejorar ventas y marketing, no solo anuncios."

## Acceptance Criteria

### AC1: Acquisition System Before Channel Recommendations

Given a user asks to improve acquisition,
when `/kokoro-acquisition-run` is invoked,
then Kokoro must first resolve the invitado and classify the request as acquisition system work, not just a platform optimization.

### AC2: Bottleneck Diagnosis

Given context or data is available,
when Kokoro analyzes the acquisition system,
then it must map the Customer Factory and Funnel stages: acquire, activate, retain, revenue, refer; plus awareness, consideration, decision, experience, and loyalty.

### AC3: Data and Tracking Gates

Given the user wants data-backed recommendations,
when Kokoro enters diagnosis,
then it must verify MCP/platform availability, data completeness, landing path, CRM/follow-up visibility, and tracking health before claiming performance causes.

### AC4: Offer and Landing Integration

Given traffic alone may not solve the acquisition constraint,
when Kokoro recommends action,
then it must include Oferta Mafia, landing decision flow, CTA clarity, friction, proof, and follow-up gaps where relevant.

### AC5: Executive Action Plan and Permission Boundaries

Given acquisition work may affect campaigns, landings, CRM, tracking, or private reports,
when Kokoro reaches recommendations,
then it must separate facts, assumptions, bottlenecks, what not to touch yet, next actions, and ask explicit permission before mutating assets, platforms, tracking, or private files.

## Scenario Examples

### Scenario: More Leads Request

Input:

```text
/kokoro-acquisition-run cliente_01 necesito mas leads
```

Expected behavior:

- Kokoro resolves the invitado.
- Kokoro checks whether the bottleneck is traffic, conversion, offer, tracking, or follow-up.
- Kokoro does not default to creating ads before diagnosing the system.

### Scenario: Data Unavailable

Input:

```text
/kokoro-acquisition-run cliente_01 ventas bajaron
```

Expected behavior:

- Kokoro verifies connected platforms and tracking.
- Kokoro marks data-backed gates as Blocked or Partial if MCP is unavailable.
- Kokoro provides a manual diagnostic path without inventing metrics.

### Scenario: Landing Bottleneck

Input:

```text
/kokoro-acquisition-run cliente_01 muchos clics pocos contactos
```

Expected behavior:

- Kokoro checks analytics and tracking first.
- Kokoro routes to landing and offer diagnosis before recommending more traffic.
- Kokoro distinguishes traffic quality from landing/CTA/follow-up friction.

# S40.9 Design: Runtime Documentation and Share Readiness

## Design Goal

Close E40 by making Kokoro safe to share and honest to install.

The output should answer three practical questions:

- What does each runtime load automatically?
- What operating runs exist now?
- What must be checked before sharing Kokoro with an invitado, collaborator, or public audience?

## Proposed Shape

Create `/kokoro-share-readiness` as the safety run that applies:

1. `kokoro-orchestrator-contract.md`
2. `kokoro-privacy-protocol.md`
3. `kokoro-share-readiness.md`
4. git status and ignore checks
5. secret/path scan
6. runtime expectation review
7. final Pass/Hold/Private Only decision

Update docs so README, AGENTS, `/kokoro`, and the executive router agree.

## Risks

| Risk | Mitigation |
|---|---|
| README overpromises runtime behavior | Replace automatic-load language with a truth table. |
| Share checklist remains hidden | Add direct command and router availability. |
| User shares private workspace state | Checklist includes git status, ignored paths, and Private Only decision. |
| No MCP demo feels broken | Document methodology-only demo path with blocked/skipped data gates. |

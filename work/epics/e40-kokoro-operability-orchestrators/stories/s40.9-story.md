# S40.9 Story: Runtime Documentation and Share Readiness

## User Story

As a founder, marketing director, sales leader, or collaborator installing Kokoro,
I want the runtime expectations and share-readiness process to be explicit,
so that I can use Kokoro safely without assuming commands, MCPs, or private data handling work the same in Claude, Codex, and Hermes.

## Context

E40 has created the operating layer: executive router, shared orchestrator contract, Google Ads run, weekly marketing pulse, creative campaign run, launch run, and acquisition run. The final story makes the system shareable by aligning README and AGENTS with real runtime behavior and by turning the existing share-readiness checklist into a callable Kokoro command.

## Acceptance Criteria

### AC1: Runtime Expectations Are Accurate

Given the README describes install and usage,
when a user reads the multi-CLI section,
then it must not claim Codex automatically loads `.claude/commands/` as slash commands, and it must explain how Claude, Codex, and Hermes differ.

### AC2: Share Readiness Is A Command

Given a user asks whether Kokoro is safe to share,
when `/kokoro-share-readiness` is invoked,
then Kokoro must load the privacy protocol and share-readiness checklist, run the public repo checks, and produce Pass, Hold, or Private Only.

### AC3: Router Availability

Given the command contract exists,
when `/kokoro` routes "share Kokoro safely" or similar requests,
then `/kokoro-share-readiness` must be marked as Available in the executive router and the main router examples.

### AC4: Final Operational Checklist

Given Kokoro may be shared with guests or collaborators,
when the docs are updated,
then they must include a concise checklist covering runtime loading, MCP boundaries, private workspace boundaries, orchestrator gates, and demo path without MCP.

### AC5: Privacy Verification

Given this is a public repo,
when the story is verified,
then the scan must show no credentials, real guest paths, private reports, generated private assets, account mappings, or newly tracked private files.

## Scenario Examples

### Scenario: Codex Install

Input:

```text
Instale Kokoro en Codex. Que hago?
```

Expected behavior:

- Docs say Codex reads `AGENTS.md`.
- Docs say Codex does not automatically expose `.claude/commands/` as native slash commands.
- Docs instruct the operator to read and invoke command markdown manually when needed.

### Scenario: Share Before Client Demo

Input:

```text
/kokoro-share-readiness antes de compartir
```

Expected behavior:

- Kokoro checks git status, secrets, ignored private paths, runtime differences, and MCP boundaries.
- Kokoro returns Pass/Hold/Private Only.
- Kokoro lists blockers before recommending sharing.

### Scenario: MCP Missing

Input:

```text
Quiero probar Kokoro sin MCP
```

Expected behavior:

- Docs explain methodology-only demo path.
- Orchestrators must mark platform data gates as Blocked or Skipped.
- Kokoro must not invent live metrics.

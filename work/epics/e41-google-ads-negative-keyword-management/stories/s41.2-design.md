# S41.2 Design: Action Contract for Delete

## Complexity

Moderate (single surface contract decision with strong backward compatibility constraints).

## Problem

Sin contrato firmado, S41.3 puede introducir regresiones en `add` o devolver respuestas inconsistentes.
`S41.1` confirma que el contrato real no está disponible en este checkout, por lo que el diseño debe mantener dos caminos de implementación segura.

## Target Contract

### Option A (preferred if contract supports explicit delete operation)

- `action`: `negative_keyword.delete`
- `text`, `match_type`, `ad_group_id`/`campaign_id`, optional `dry_run`
- Response: `status`, `requested`, `removed`, `not_found`, `errors`, `audit_event_id`
- Estado final físico o equivalente por API del connector.

### Option B (fallback)

- Reusar acción de gestión existente con `action: mutate_negative_keywords` + `operation: delete` o `op: delete` + campo `state: removed`.
- No cambiar forma de `add`.
- Documentar bandera de compatibilidad y validación explícita.

## Decision Rule

Usar Option A cuando el conector confirmado exponga delete nativamente.
Usar Option B si el conector solo soporta add/lifecycle activo.

## Compatibility Rules

- No romper firmas existentes de `add`.
- `delete` debe ser idempotente por (`text`,`match_type`,`scope`).
- Respuesta parcial debe ser válida (`requested > removed`, con detalle de `not_found` y `already_removed`).

## Test Expectations

- Caso feliz (eliminación simple).
- `not_found` (cero coincidencias en estado activo).
- `already_removed` (estado previo de removido).
- Error parcial por permisos/API.

## Success Definition

- S41.3 puede implementarse únicamente con este diseño y el contrato confirmado por S41.1.

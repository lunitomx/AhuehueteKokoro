# S41.2 Story: Action Contract for Delete

## User Story

Como operador de Kokoro,
quiero un contrato explícito para `negative_keyword.delete`,
para invocar limpieza sin romper el contrato actual de `add` ni la trazabilidad de bitácora.

## Acceptance Criteria

### AC1: Contrato de entrada definido

El contrato de `negative_keyword.delete` debe quedar explicitado con:

- `action` o campo equivalente de intención de borrado
- `text`
- `match_type` (`EXACT|PHRASE|BROAD` o alternativa del API real)
- `ad_group_id` o `campaign_id` según corresponda
- `scope` opcional (incluye `dry_run` cuando aplique)
- campo de `state` objetivo (`removed`/`inactive` o equivalente)

### AC2: Contrato de salida definido

Respuesta de operación debe incluir:

- estado general (`success|partial|failed`)
- número de solicitados
- elementos removidos/actualizados
- no encontrados o ya removidos
- errores por elemento
- `audit_event_id` o equivalente de traza

### AC3: Compatibilidad

Debe documentar explícitamente cómo convive con `add`:

- no romper payloads actuales de `add`
- no cambiar campos obligatorios de `add`
- permitir fallback (`removed` lógico) cuando no exista borrado físico nativo.

### AC4: Dependencia de fuente real

`S41.2` no se mueve a implementación hasta que S41.1 confirme la localización exacta y schema real del connector (no solo issue/docs).

# S42.1 Retrospective: Learning Record Schema

## What was implemented

Created `.claude/knowledge/kokoro-google-ads-learning-record.md` — the
canonical schema for Google Ads learning entries within the Kokoro
Session Log.

The schema defines:
- 8 core fields (4 required, 4 optional) shared across all Kokoro sessions
- 10 Google Ads-specific optional fields (platform, campaign_type, task_group,
  task, cadence, landing_page, asset_group, change_made, reason, learning_state)
- Backward compatibility: all new fields are optional, no existing fields
  renamed or removed
- Mapping to the Learning Dashboard's three layers (Plan/Ejecución/Aprendizaje)

| Aligns with session-log vocabulary | ✅ | Uses same field names and types as `kokoro-session-log.md` |

## What went well

- The existing `session_log` already had `platform`, `campaign_type`, and
  `learning_state` — the schema formalized what was implicit.
- The Learning Dashboard's three-layer model made field categorization
  straightforward: operational fields → Ejecución, insights → Aprendizaje,
  cadence → Plan.
- Keeping all fields optional avoided a migration burden.

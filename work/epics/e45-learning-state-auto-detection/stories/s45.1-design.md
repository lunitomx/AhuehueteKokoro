# S45.1 Design: Meta Ads Learning State Detector

## Current Surface

`learning_state` es un campo manual en el session_log. El operador decide
si escribe "learning", "stable" o "needs_attention". No hay automatización.

## Target Behavior

Al cerrar una sesión de Meta Ads (`/kokoro-ads` o `/kokoro-close`),
Kokoro consulta el MCP de Meta Ads (si está disponible) y aplica los
thresholds definidos en S45.1 para inferir learning_state automáticamente.

## Detector Contract

### Input
- `campaign_id`: ID de la campaña en Meta Ads
- `mcp_available`: boolean — ¿el MCP respondió?

### Output
- `learning_state`: "learning" | "stable" | "needs_attention" | null
- `reason`: string — explicación de por qué se infirió ese estado
- `source`: "mcp" | "manual"

### Decision Tree

```
mcp_available?
├── NO → return null (sin learning_state)
└── SÍ → fetch campaign data
    ├── campaign_age < 7 days → "learning" (razón: campaña nueva)
    ├── conversions_7d < 50 → "learning" (razón: volumen insuficiente)
    ├── cpa_change > +30% → "needs_attention" (razón: CPA subió X%)
    ├── spend > 0 AND conversions_7d == 0 → "needs_attention" (razón: sin conversiones)
    └── else → "stable" (razón: CPA estable en ±15%)
```

### Fallback
Si el MCP no responde o no está configurado → learning_state = null.
Nunca se inventa un valor.

### Override
El operador puede escribir manualmente learning_state en el session_log.
Si existe un valor manual, el detector NO lo sobrescribe — el manual
tiene prioridad sobre el automático.
